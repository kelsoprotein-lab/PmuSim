# PmuSim - PMU 主站模拟器设计文档

## 概述

PmuSim 是一个 PMU（相量测量单元）主站模拟器/测试工具，用于在没有真实 PMU 数据集中器的情况下，模拟主站行为来测试子站（一次调频系统）。

- **角色**：主站（数据集中器）
- **协议**：同时支持金风 PMU 协议 V2 和 V3，界面可切换
- **规模**：支持 2~10 个子站并发连接
- **界面**：Tkinter 桌面 GUI
- **语言**：Python，零第三方依赖（仅标准库）

## 协议规范摘要

基于金风科技 PMU 协议 V2（Q/GDW 131）和 V3（GB/T 26865.2）版报文解析指导手册。

### 帧类型

| 帧类型 | SYNC Bit6~4 | 方向 | 用途 |
|--------|------------|------|------|
| 数据帧 DataFrame | 000 | 子站→主站 | 实时监测数据（模拟量、开关量） |
| 配置帧 CFG-1 | 010 | 子站→主站 | 子站硬件配置和数据采集能力 |
| 配置帧 CFG-2 | 011 | 双向 | 数据帧的具体格式和传输规则 |
| 命令帧 Command | 100 | 双向 | 控制命令（召唤配置、开启数据、心跳等） |

### V2 与 V3 关键差异

| 特性 | V2 | V3 |
|------|----|----|
| 版本位 SYNC Bit3~0 | 0010 | 0011 |
| 帧字段顺序 | SYNC-SIZE-SOC-IDCODE-payload-CHK | SYNC-SIZE-IDCODE-SOC-FRACSEC-payload-CHK |
| 命令帧大小 | 20 字节 | 24 字节（含 FRACSEC） |
| FRACSEC 时间质量 | 仅数据帧有 | 所有帧均有，Bit27~24 定义时钟状态 |
| 默认端口 | 管理 7000 / 数据 7001 | 管理 8000 / 数据 8001 |
| 确认命令 | Bit15~13=111 "确认命令" | 111="肯定确认"，001="否定确认" |

### CMD 命令字定义

| 命令 | Bit 值 | 说明 |
|------|--------|------|
| 关闭实时数据 | Bit3~0 = 0001 | 主站下发 |
| 打开实时数据 | Bit3~0 = 0010 | 主站下发 |
| 发送头文件 | Bit3~0 = 0011 | 主站下发 |
| 发送 CFG-1 | Bit3~0 = 0100 | 主站下发 |
| 发送 CFG-2 | Bit3~0 = 0101 | 主站下发 |
| 心跳信号 | Bit15~13 = 010 | 双向 |
| 下传 CFG-2 | Bit15~13 = 100 | 主站下发 |
| 肯定确认 | Bit15~13 = 111 | 子站响应 |
| 否定确认 | Bit15~13 = 001 | 子站响应（仅 V3） |

### 通信流程

```
主站                                子站
 │                                   │
 │──── 召唤 CFG-1 (CMD=0x0004) ────►│
 │◄─── 上传 CFG-1 ─────────────────│
 │──── 下传 CFG-2 命令 (CMD=0x8000)─►│
 │◄─── 确认 (CMD=0xE000) ──────────│
 │──── 下传 CFG-2 配置帧 ──────────►│
 │◄─── 确认 (CMD=0xE000) ──────────│
 │──── 召唤 CFG-2 (CMD=0x0005) ────►│
 │◄─── 上传 CFG-2 ─────────────────│
 │──── 开启数据 (CMD=0x0002) ──────►│
 │◄─── 确认 (CMD=0xE000) ──────────│
 │◄─── 数据帧 (持续) ──────────────│
 │──── 心跳 (CMD=0x4000) ─────────►│
 │◄─── 心跳 (CMD=0x4000) ──────────│
```

### 配置帧关键字段

- **STN**: 站名，16 字节，汉字或 ASCII，不足前补 "0"
- **IDCODE**: PMU 标识符，8 字节 ASCII
- **FORMAT**: 数据格式（极坐标/直角坐标、整数/浮点）
- **PHNMR/ANNMR/DGNMR**: 相量/模拟量/开关量数量
- **CHNAM**: 通道名称，每个 16 字节
- **ANUNIT**: 模拟量转换因子，4 字节/通道（V2: 原值 * 0.01 等；V3: 原值 * 0.00001）
- **DIGUNIT**: 开关量掩码（高字节：常开/常闭；低字节：有效位）
- **FNOM**: 额定频率（50Hz）
- **PERIOD**: 传送周期，基波倍数 * 100，换算为毫秒

### 数据帧关键字段

- **FRACSEC**: 秒等分，配合 MEAS_RATE 计算毫秒时标
- **STAT**: 状态字（数据可用性、同步状态、触发信息等）
- **ANALOG**: 原始整数值，需结合 ANUNIT 转换因子计算实际物理量
- **DIGITAL**: 开关量位图，低位到高位对应通道顺序

## 架构设计

### 三层架构

```
┌─────────────────────────────────────────────┐
│              Tkinter UI (主线程)              │
│  ┌──────────┬──────────┬──────────────────┐  │
│  │ 连接管理  │ 配置查看  │  实时数据监控     │  │
│  └──────────┴──────────┴──────────────────┘  │
│         ▲ root.after() 轮询 Queue            │
├─────────┼───────────────────────────────────┤
│         │      queue.Queue (线程安全)         │
├─────────┼───────────────────────────────────┤
│         ▼                                    │
│       asyncio 事件循环 (后台线程)              │
│  ┌─────────────────────────────────────────┐ │
│  │         MasterStation                   │ │
│  │  ┌───────────┐  ┌───────────┐          │ │
│  │  │SubStation │  │SubStation │  ...      │ │
│  │  │Session    │  │Session    │          │ │
│  │  │ ├管理TCP  │  │ ├管理TCP  │          │ │
│  │  │ └数据TCP  │  │ └数据TCP  │          │ │
│  │  └───────────┘  └───────────┘          │ │
│  └─────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────┐ │
│  │      协议层 (V2/V3 编解码)               │ │
│  │  FrameParser / FrameBuilder / CRC16     │ │
│  └─────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
```

1. **UI 层** — Tkinter 界面，展示状态和数据，发送用户操作指令
2. **网络层** — asyncio 管理所有 TCP 连接，每个子站两条连接（管理管道 + 数据管道）
3. **协议层** — 纯二进制报文编解码，不涉及 I/O，支持 V2/V3 切换

层间通过 `queue.Queue` 解耦。

### 连接模型

主站作为 TCP Server 监听两个端口：

- **管理管道端口**（V2 默认 7000，V3 默认 8000）：传输命令帧和配置帧
- **数据管道端口**（V2 默认 7001，V3 默认 8001）：接收子站数据帧

同一子站的两条连接通过 IDCODE 进行配对。

## 协议层设计

### 帧公共头部

```python
@dataclass
class FrameHeader:
    sync: int          # 0xAA + type/version
    frame_size: int    # 帧总长度
    idcode: str        # 8 字节 ASCII
    soc: int           # 世纪秒
    fracsec: int       # 秒等分（V3 所有帧，V2 仅数据帧）
```

V2/V3 字段顺序差异由解析器内部处理：
- V2: SYNC - SIZE - SOC - IDCODE - payload - CHK
- V3: SYNC - SIZE - IDCODE - SOC - FRACSEC - payload - CHK

### 核心类

- **`FrameParser`** — 输入 `bytes`，输出帧对象。自动从 SYNC 低 4 位识别 V2/V3，从 Bit6~4 识别帧类型，校验 CRC16
- **`FrameBuilder`** — 输入帧对象，输出 `bytes`。根据版本决定字段排列，自动计算 FRAMESIZE 和 CRC16
- **`CRC16`** — CRC-CCITT 校验算法
- **`AnalogConverter`** — 结合 ANUNIT 转换因子将原始整数转为实际物理量

### V2/V3 差异处理

不使用继承体系，用 `version` 字段（2 或 3）控制编解码时的字段顺序和是否包含 FRACSEC。差异点集中在 Parser 和 Builder 内部的条件分支。

## 网络层设计

### SubStationSession

```python
@dataclass
class SubStationSession:
    idcode: str                    # 子站标识
    version: int                   # 2 或 3，从 SYNC 自动识别
    mgmt_reader: StreamReader      # 管理管道读
    mgmt_writer: StreamWriter      # 管理管道写
    data_reader: StreamReader      # 数据管道读
    data_writer: StreamWriter      # 数据管道写
    cfg1: ConfigFrame | None       # 已接收的 CFG-1
    cfg2: ConfigFrame | None       # 当前生效的 CFG-2
    state: SessionState            # 状态机
    last_heartbeat: float          # 最近心跳时间
```

### 状态机

```
CONNECTED → CFG1_RECEIVED → CFG2_SENT → STREAMING
    │              │              │            │
    └──────────────┴──────────────┴──── 断开 → DISCONNECTED
```

- 用户可在界面上手动触发每一步
- "一键握手"按钮自动走完整个流程
- 心跳由后台定时任务发送

### MasterStation

```python
class MasterStation:
    mgmt_server: asyncio.Server
    data_server: asyncio.Server
    sessions: dict[str, SubStationSession]
    event_queue: queue.Queue       # → UI
    cmd_queue: asyncio.Queue       # ← UI
```

### 子站配对逻辑

同一子站先后建立两条连接，通过首帧中的 IDCODE 进行配对，合并到同一个 SubStationSession。设置 10 秒配对超时。

### 标签页内容

**配置页**：展示选中子站的 CFG-1/CFG-2 解析结果
- 站名、IDCODE、协议版本、FORMAT
- 模拟量通道列表（名称 + 转换因子）
- 开关量通道列表（名称 + 常开常闭 + 有效位）
- 传送周期（PERIOD → 毫秒）
- 下传 CFG-2 前可编辑 PERIOD

**实时数据页**：表格滚动刷新
- 每行：时间戳 | 模拟量 1~N（转换后实际值）| 开关量状态 | STAT 标志
- UI 每 200ms 批量刷新一次

**通信日志页**：收发报文流水
- 每行：时间 | 方向(→/←) | 帧类型 | 关键摘要
- 保留最近 1000 条

### UI 与后台交互

- **UI → 后台**：`cmd_queue` 写入指令，如 `("send_cfg1_request", idcode)`
- **后台 → UI**：`event_queue` 写入事件，如 `("cfg1_received", idcode, cfg1_obj)`
- `root.after(50, poll_queue)` 每 50ms 轮询

## 错误处理

### 连接异常

- **子站断开**：session 置为 DISCONNECTED，UI 更新为"离线"，清理连接
- **单管道超时**：10 秒内另一条管道未建立则标记异常
- **IDCODE 配对失败**：保持未配对状态，日志提示

### 协议容错

- **CRC 校验失败**：丢弃帧，日志记录
- **未知帧类型**：跳过，日志记录原始十六进制
- **帧长度不符**：丢弃，日志警告
- **TCP 粘包/拆包**：先读 4 字节（SYNC + SIZE）获取总长，再读取剩余字节

### 心跳机制

- 主站每 30 秒发送心跳（可配置）
- 连续 3 次无响应标记子站离线

### UI 性能

- 数据帧最高 100 帧/秒，UI 每 200ms 批量刷新
- 通信日志保留最近 1000 条

### 依赖

仅 Python 标准库：`tkinter`、`asyncio`、`struct`、`threading`、`queue`、`dataclasses`、`enum`。零第三方依赖。

### 设计约束

- `protocol/` 纯函数/数据类，不引用 asyncio 或 tkinter，可独立单元测试
- `network/` 只依赖 protocol，不引用 tkinter
- `ui/` 只依赖 protocol 数据类用于展示，不直接操作 socket
- 三层通过 Queue 通信，无循环依赖
